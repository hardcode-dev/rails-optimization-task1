# Case-study оптимизации

## Актуальная проблема
В нашем проекте возникла серьёзная проблема.

Необходимо было обработать файл с данными, чуть больше ста мегабайт.

У нас уже была программа на `ruby`, которая умела делать нужную обработку.

Она успешно работала на файлах размером пару мегабайт, но для большого файла она работала слишком долго, и не было понятно, закончит ли она вообще работу за какое-то разумное время.

Я решил исправить эту проблему, оптимизировав эту программу.

## Формирование метрики
Для того, чтобы понимать, дают ли мои изменения положительный эффект на быстродействие программы я придумал использовать такую метрику: время работы программы.

## Гарантия корректности работы оптимизированной программы
Программа поставлялась с тестом. Выполнение этого теста в фидбек-лупе позволяет не допустить изменения логики программы при оптимизации.

## Feedback-Loop
Для того, чтобы иметь возможность быстро проверять гипотезы я выстроил эффективный `feedback-loop`, который позволил мне получать обратную связь по эффективности сделанных изменений за несколько секунд. 

Вот как я построил `feedback_loop`:
1. Создал вспомогательный файл `temp_data.txt`.
2. В методе `work` заменил файл, из которого происходит чтение с `data.txt` на `temp_data.txt`.
3. На время закомментировал блок кода с `minitest`, т.к. там происходит запись в файл `data.txt`.
4. Создал файл `spec.rb` с кодом по примеру того, что был в вебинаре -- для проверки того, выполняется ли программа за заданное время. Уставку времени поставил заведомо невыполнимую (1 мс) -- для получения информации о затраченном времени в отчёте `rspec` о прошедших тестах. В начале файла `spec.rb`, до начала блока с тестами, происходило отключение GarbageCollector (`GC.disable`), в конце, после блока с тестами -- включение (`GC.enable`) (чтобы не забыть и не оставить его потом в таком положении) и его запуск (`GC.start`), потому как не был уверен, запускается ли он только лишь при включении.
5. Добавил в блок с тестами тесты на характеристику асимптотики. Для работы `perform_power` временно изменил алгоритм чтения из файла в методе `work` на `File.foreach('temp_data.txt').first(n)`, добавив `n` в параметры, принимаемые функцией. Никакие из асимптотических тестов, кроме того, что на `perform_power`, не прошли. Удалил асимптотические тесты.
6. С помощью команд вида `head -n 10000 data_large.txt > temp_data.txt` заполнял вспомогательный файл заданным количеством строк (10, 100, 1_000, 10_000, 100_000).
7. Проверил время работы программы при каждом из этих значений с помощью отчёта `rspec` о непройденном тесте:
`    10 lines: performed above 1.5 ms (± 25.1 μs);
    100 lines: performed above 9.6 ms (± 133 μs);
  1_000 lines: performed above 199 ms (± 2.4 ms);
 10_000 lines: performed above 16.2 sec (± 111 ms);
100_000 lines: результата за полчаса не дождался, завершил выполнение программы;`
8. С помощью отношения `n = log(t2/t1)`, где `n` -- показатель степени, `t1` и `t2` -- время выполнения программы для "соседних" значений объёмов входных данных, вычислил значение степени в степенной (по предположению `rspec-benchmark`) функции для пар "соседних" значений объёмов входных данных:
`   100 lines: 0.81;
  1_000 lines: 1.32;
 10_000 lines: 1.91;
100_000 lines: предположив, что рост значения степени пойдёт примерно с той же скоростью, принял n = 2.6, расчётное время выполнения получилось около 108 минут -- сделал вывод, что поступил правильно, не став дожидаться окончания выполнения теста на 100_000 строк;`
Ввиду роста показателя степени с ростом объёма входных данных, сделал вывод, что функция асимптотики сложнее степенной.
9. Для `feedback-loop` выбрал тест для значения объёма входных данных в 10_000 строк, как наибольший из "вменяемых" по времени ответа.
